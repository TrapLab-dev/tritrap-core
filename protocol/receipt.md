# Receipt Protocol

A Receipt is the artifact generated by an execution lane after completing a workload authorized by a Permit.

Receipts serve as proof that execution occurred and provide the data required for verification and arbitration.

Each lane independently produces a receipt describing the execution outcome.

---

## Receipt Structure
Receipt {
receipt_id
permit_id
lane_id
workload_hash
output_hash
execution_timestamp
runtime_metadata
signature
}

---

## Field Descriptions

**receipt_id**

Unique identifier for the receipt.

**permit_id**

Reference to the permit that authorized execution.

**lane_id**

Identifier of the execution lane producing the receipt.

**workload_hash**

Hash of the executed workload.

**output_hash**

Hash representing the execution result.

**execution_timestamp**

Time the workload completed.

**runtime_metadata**

Optional execution metadata such as model version, runtime configuration, or hardware lane.

**signature**

Lane signature verifying receipt authenticity.

---

## Receipt Generation

After execution completes, the lane must:

1. Generate output hash
2. Populate receipt fields
3. Sign the receipt
4. Forward receipt to the verification layer

---

## Role in Execution Flow

Receipts provide the evidence required for verification.

The verification layer evaluates one or more receipts produced by execution lanes and determines the accepted output.
