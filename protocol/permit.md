# Permit Protocol

The Permit is the authorization artifact that allows a workload to be executed within the TriTrap system.

TriTrap operates under a **deny-by-default execution model**. No inference workload may be executed without a valid permit issued by a Gate node.

A Permit defines:

• what workload may be executed
• where execution may occur
• when execution is valid
• how results will be verified

Permits are issued by the Gate node after validating an incoming request.

---

## Permit Structure

A Permit contains the following logical fields:
Permit {
permit_id
workload_hash
permitted_lane
issue_timestamp
expiration_timestamp
verification_policy
signature
}

### Field Descriptions

**permit_id**

Unique identifier for the permit issued by the Gate.

**workload_hash**

Hash representing the authorized workload or inference request.

**permitted_lane**

Identifier of the execution lane allowed to process the workload.

**issue_timestamp**

Time at which the permit was generated.

**expiration_timestamp**

Time after which the permit becomes invalid.

**verification_policy**

Defines how execution results will be verified by the verification layer.

**signature**

Cryptographic signature generated by the Gate node to ensure permit authenticity.

---

## Permit Validation

Before execution begins, the execution lane must validate the permit:

1. Verify the Gate signature
2. Confirm permit has not expired
3. Confirm workload hash matches request
4. Confirm lane authorization

If validation fails, the workload must not execute.

---

## Security Model

Permits enforce TriTrap's **authorization-first execution model**.

Execution lanes are not trusted to initiate workloads outside permit authority.
All execution must originate from a validated permit issued by the Gate.

This separation ensures that:

• authorization decisions are centralized
• execution environments remain controlled
• unauthorized workloads cannot execute

---

## Role in Execution Flow

Within the TriTrap execution model:

1. Client submits request
2. Gate validates request
3. Gate issues Permit
4. Permit authorizes lane execution
5. Lane produces execution receipt
6. Verification layer validates results
