# Contact and Review

This document explains how reviewers, partners, and technical readers should approach the TriTrap public repository.

The public repository is the architecture and review front door. It is not a deployment manual, proof archive, or private runtime export.

---

## Review Scope

Public review may cover:

- governed execution architecture
- permit-scoped workload authorization
- policy-defined verification
- heterogeneous execution lane concepts
- receipt-backed evidence concepts
- deny-by-default execution
- governed compute commitment concepts
- transaction-bound telemetry concepts
- public claim boundaries and non-disclosure posture

Public review should not require access to private runtime mechanics.

---

## Controlled Diligence

Stronger evidence, where appropriate, belongs in controlled review rather than the public repository.

Controlled diligence may discuss evidence classes, maturity posture, and review process without publishing private deployment material. The public repository may describe that controlled review exists, but it does not promise open access to private runtime systems.

---

## Contact Routing

Use `traplab.dev` as the technical and reviewer-facing project surface.

Use `mntdintel.com` as the outward business, brand, and operations identity surface.

Private infrastructure endpoints, local services, dashboards, and internal operator tools are not public product endpoints and should not be requested through this repository.

---

## Boundary for Requests

Acceptable public requests include:

- clarification of public architecture wording
- questions about claim boundaries
- documentation consistency issues
- security-boundary concerns in public material
- partner or reviewer routing questions

Requests for private topology, runtime access, proof bundles, keys, receipt material, operational scripts, infrastructure configuration, or deployment instructions are outside the public repository scope.
