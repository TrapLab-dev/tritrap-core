# Public Roadmap

This roadmap is public-safe. It describes the direction of TriTrap documentation and review posture without publishing private runtime mechanics.

It is not a deployment plan, release commitment, or proof archive.

---

## Current Public Foundation

The public repository currently documents:

- governed execution architecture
- deny-by-default execution
- permit-scoped authorization
- separated Gate, execution lane, and verification responsibilities
- receipt-backed evidence concepts
- policy-defined arbitration
- fail-closed validation posture
- public claim boundaries
- controlled diligence posture

---

## Public-Safe Alignment Direction

Near-term public documentation should continue to clarify:

- governed compute commitment as an architecture concept
- transaction-bound telemetry as an evidence-binding concept
- heterogeneous verification as stronger than identical-lane mirroring
- specialized or accelerator-backed execution as private implementation detail unless separately disclosed
- private runtime evidence as withheld by design
- reviewer paths that do not expose deployment mechanics

---

## What Remains Private

The public roadmap does not publish:

- private runtime proof
- private evidence archives
- infrastructure configuration
- service definitions
- runtime paths
- key material
- receipt or permit material
- operational commands
- private topology
- deployment scripts

---

## Public Documentation Backlog

Planned public-safe improvements:

1. Keep claim boundaries synchronized with current public posture.
2. Maintain reviewer-friendly navigation and one-page summaries.
3. Add business and reviewer contact routing.
4. Keep issue and pull-request templates aligned to the disclosure boundary.
5. Add public examples only when they are synthetic, redacted, or intentionally released.
6. Continue separating public architecture from controlled private diligence.

No timeline is implied by this roadmap.
